1 Project Overview

This thesis project is a comprehensive HR Attendance Management System designed to solve real-world employee attendance tracking challenges. It was built by SnoopyGenZ as a full-stack, multi-platform application.

Problem Statement

Manual attendance tracking is prone to errors, time fraud, and lacks real-time visibility. This system automates the entire process with GPS verification, QR scanning, AI-powered risk prediction, and multi-platform access.

What the System Does

  • Check-in/Check-out with GPS geofencing (location verified within 100 meters)
  • QR Code scanning for location-based attendance verification
  • Leave/Permission management with approval workflow
  • Payroll processing integrated with attendance records
  • Employee evaluation system
  • Company asset management with stock tracking
  • Accident expense reporting
  • Mission letter system with digital signatures
  • Real-time notifications via Socket.IO, Firebase, and Telegram
  • AI-powered attendance risk prediction
  • Bilingual support (English and Khmer)

Key Architecture Decisions

  • Generic CRUD Routes - A single dynamic route handles all MongoDB collections, reducing code duplication across 60+ models
  • Socket.IO Broadcasting - Any data change triggers a real-time event to all connected clients, which then selectively refresh affected data
  • Server Timestamp - All time-sensitive operations use server time (Asia/Phnom_Penh timezone) to prevent client-side time manipulation
  • Multi-Firebase Projects - Three separate Firebase projects for web admin, web check-in, and mobile app push notifications

5 Backend (Node.js API)

Entry Point: app.js (572 lines)

The main server file that orchestrates everything:

  • Connects to MongoDB (hrsystem database, maxPoolSize: 200)
  • Creates HTTP server on port 4000
  • Configures CORS, Helmet.js security headers, JSON payload limits (1GB)
  • Sets up rotating file logging (daily rotation, gzip, 14-day retention)
  • Initializes Socket.IO for real-time communication
  • Starts Telegram bots per branch for notifications
  • Uses node-cron to auto-update meeting statuses every 5 minutes
// app.js - Simplified structure const express = require('express'); const mongoose = require('mongoose'); const { Server } = require('socket.io'); // Connect to MongoDB mongoose.connect('mongodb://localhost:27017/hrsystem', { maxPoolSize: 200 }); // Mount routes app.use('/loan/api', authRoutes); // Authentication app.use('/loan/api', dynamicRoutes); // Generic CRUD (60+ models) app.use('/loan/api', timestampRoutes); // Server time app.use('/loan/api', telegramRoutes); // Telegram bots app.use('/loan/api', evaluationRoutes);// Evaluation stats app.use('/loan/api', predictionRoutes);// AI prediction // Start server on port 4000 httpServer.listen(4000);

Core Route: dynamicRoute.js - Generic CRUD

Instead of creating separate routes for each of the 60+ MongoDB collections, a single dynamic route handles all operations by accepting collection names as parameters:

// dynamicRoute.js - Key endpoints router.post('/insertDoc/:collectionName', // Create router.post('/updateDoc/:collectionName', // Update router.post('/deleteDoc/:collectionName', // Delete router.get('/getAllDocs/:collectionName', // Read (search, filter, sort) router.get('/getPagination/:collectionName', // Paginated read router.post('/getNextId/:collectionName/:branchId', // Sequential ID router.post('/pushNotification', // Firebase push router.post('/extractTextFromImage', // OCR (ID cards) router.post('/verifyQrCode', // QR + GPS verify

Key Controllers

Authentication (authController.js - 411 lines)

  • Login: Validates credentials with bcrypt, implements progressive account lockout (5 failed attempts = lock, duration increases)
  • OTP: Generates 6-digit code, stores with 5-minute expiry, sends via email
  • JWT Token: Issues tokens on successful login/OTP verification

Data Operations (useDataController.js - 1400+ lines)

  • Maintains a modelMap mapping 60+ collection names to Mongoose models
  • Custom ID Generation: Branch-scoped sequential IDs (e.g., BR000001)
  • Auto Password Hashing: Automatically hashes passwords for User collection
  • Socket.IO Broadcasting: Emits events on any data mutation

Query Engine (getDataController.js - 1500+ lines)

  • Supports 10+ query operators: ==, !=, >, <, >=, <=, arrayContains, in, notIn, regex
  • Date-range conditions with date field filtering
  • Multi-field regex search
  • Dynamic population (MongoDB joins)
  • Pagination with skip/limit

OCR Vision (visionController.js - 263 lines)

  • Uses Google Cloud Vision API to extract text from Cambodian national ID cards
  • Parses Khmer script, extracts ID number, name (Khmer + English), DOB, gender, address
  • Supports MRZ (Machine Readable Zone) parsing

QR Location Verification (qrController.js - 69 lines)

  • Uses the Haversine formula to calculate distance between GPS coordinates
  • Returns valid: true if employee is within 100 meters of a registered location
// Haversine formula for distance calculation function haversineDistance(lat1, lon1, lat2, lon2) { const R = 6371000; // Earth radius in meters const dLat = (lat2 - lat1) * Math.PI / 180; const dLon = (lon2 - lon1) * Math.PI / 180; const a = Math.sin(dLat/2) * Math.sin(dLat/2) + Math.cos(lat1 * Math.PI / 180) * Math.cos(lat2 * Math.PI / 180) * Math.sin(dLon/2) * Math.sin(dLon/2); return R * 2 * Math.atan2(Math.sqrt(a), Math.sqrt(1 - a)); } // Distance < 100 meters = valid check-in location
